# PR 裁き（レビュー会）

## 手順

1. `gh pr list --state open --json number,title,author,createdAt,baseRefName,isDraft` で一覧化
2. **捌く順序**を決めて表で提示:
   1. 他のPRをブロックしているもの（同ファイルを触る後続がある）
   2. 小さいもの（差分小 → 即判断できる）
   3. 古いもの（コンフリクトが育つ前に）
   - Draft は対象外（一覧には載せる）
3. 各PRについて機械チェック → 内容レビュー:
   - CI 状態: `gh pr checks <n>`
   - base が staging か（main 直 PR は原則差し戻し）
   - `/review3` の3視点レビューを実施
4. **マージ判断基準**（全部満たしたらマージ可）:
   - [ ] CI green
   - [ ] review3 で High 指摘なし（Mid は別issue化で通過可）
   - [ ] PR の内容が issue のスコープ内（スコープクリープなし）
   - [ ] DB変更を含むなら `/db-change` の手順が先行済み
5. 判断結果を1PRずつ実行:
   - マージ → `gh pr merge <n> --squash`（マージ方式は既存慣習に合わせる）
   - 差し戻し → 下の型でコメント
6. 会の最後にサマリ表: マージ済み / 差し戻し / 保留（理由つき）

## 差し戻しコメントの型

```
## 差し戻し理由
- [High] <何が> <なぜダメか> (<file:line>)

## 修正方針
- <どう直すか。具体的に>

## 再レビュー条件
- <これが満たされたら再度レビューする>
```

トーンは事実ベース・非難しない。良い点が1つでもあれば先に書く。

## 注意

- マージは必ずユーザーの同席時 or 明示許可の範囲で。まとめて無言マージしない
- マージ後に staging へ push が積まれたら `/smoke` を出す
