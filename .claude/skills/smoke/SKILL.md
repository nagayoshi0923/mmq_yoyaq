# 動作確認チェックリスト生成

## 対象の特定

- 引数があればそれを使う（例: `HEAD~3..HEAD`、`#201`）
- 省略時は直近に staging へ push したコミット群（`git log origin/main..origin/staging --oneline` から今回分）
- `git diff --stat` と diff 本文を読み、**ユーザーが実際に触れる画面・操作**を特定する

## 絶対ルール

- **各項目に必ず「どの画面か」を書く**: メニュー名・ページ名・タブ名・たどり方。
  ユーザーが迷わず再現できない項目は書き直す。
  - ❌ 「予約編集が動くこと」
  - ✅ 「【予約管理 > 予約者タブ】予約をクリック → 人数を変更して保存できること」
- リグレッション項目は、変更したファイルの**呼び出し元**と**共通ロジックを使う隣接機能**を grep で特定して列挙する
- 純粋なリファクタリングでも「挙動が同じこと」の確認項目を出す
- Edge Function の発火が期待される操作（メール送信・Discord通知等）は確認項目に含める
- マイグレーションを含む場合は DB 確認項目（`npm run db:status`、対象テーブルの確認クエリ）を追加する

## 出力形式

```
## ✅ 動作確認チェックリスト

### 基本確認
- [ ] ステージング環境にデプロイされている（staging URL を開く）
- [ ] `npm run db:status` で未適用マイグレーションがないこと（マイグレーションがある場合のみ）

### 変更箇所の動作確認
- [ ] 【ページ名 > タブ名】操作 → 期待結果

### リグレッション確認
- [ ] 【ページ名】隣接機能が壊れていないこと

### 確認完了後
- [ ] 問題なければ main への反映を依頼（→ /deploy）
```

- ステージング URL: `https://mmq-yoyaq-git-staging-nagayoshi0923s-projects.vercel.app`
- ローカル確認の場合の起動コマンドも添える: `npm run dev:vercel`（http://localhost:5173）
