# バグ報告 → issue 変換

**Discord経由の自動整形（.github/issue-groomer-prompt.md）のローカル手動版。型は同一。**
型の正は `docs/templates/issue-format.md` の [Bug] セクション。

## 手順

1. `docs/templates/issue-format.md` を読む
2. 報告文から埋められる欄を埋める:
   - 概要（症状1〜2行）/ 再現手順 / 期待する動作 / 実際の動作 / 環境・補足
   - **わからない欄は「確認中」と明記**。推測で埋めない
3. 原因不明なら「## 調査してほしいこと」に `- [ ]` チェックリストを追加
   （どのテーブル・どの画面・どのログを見れば切り分くか）
4. **ヒアリング質問を最大3つ**生成。各質問に「なぜ聞くか」を一言添える。
   報告者（スタッフ/顧客）が答えやすい言葉で書く
5. 元の報告文は `<details><summary>元の報告</summary>` に一字一句そのまま保存
6. プレビューをユーザーに見せてから `gh issue create`、ラベル（bug + needs-info）付与

## 切り分けの初手（質問を作るときの観点）

- 特定データだけか、全体か（→データ不整合 vs ロジックバグ）
- いつからか（→直近デプロイとの相関。`git log origin/main --oneline -10` で確認）
- どの画面・どの操作経路か（顧客側/管理側、スマホ/PC）
- 環境は本番か staging か（staging はミラー特有の罠あり: auth orphan、app_config）

## 注意

- 「@claude」という連続文字列を出力に入れない
- 予約・決済・認証まわりのバグは委任判定を「人間主導」にする
