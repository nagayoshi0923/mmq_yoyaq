# リリースノート生成

## 手順

1. 対象範囲を特定:
   - 引数があれば従う（期間 or PR番号リスト or タグ範囲）
   - 省略時: 前回の本番反映以降にマージされたPR
     `gh pr list --state merged --base main --json number,title,mergedAt` +
     `git log <前回デプロイ>..origin/main --oneline`
2. PRを分類: 新機能 / 改善 / 不具合修正 / 内部変更（リファクタ・CI等）
3. **2種類の文面**を生成:

### スタッフ向け（運用影響ベース）

```
## 📢 システム更新のお知らせ (YYYY/MM/DD)

### 新機能・変更
- 【どの画面】何ができるようになったか。操作が変わる場合は旧→新を明記
### 不具合修正
- 【どの画面】何が直ったか（報告してくれた人がいれば伝わる書き方で）
### 注意
- 運用で気をつけること（あれば）
```

- どの画面か（ページ名・タブ名）を必ず書く
- 内部変更は「安定性向上」程度に丸めるか省略

### お客さん向け（ベネフィットベース）

```
【アップデートのお知らせ】
・「〜できるようになりました」形式。専門用語なし
・不具合修正は「予約画面の表示の不具合を修正しました」程度に丸める
・内部変更は書かない
```

- トーン: 丁寧・簡潔・絵文字は控えめ（見出しに1つ程度）
- 顧客に関係ない変更しかない場合は「お客さん向けは今回不要」と判断してよい

4. 両方をユーザーに提示。配信はユーザーが行う（Discord/LINE/メールへの投稿はしない）
