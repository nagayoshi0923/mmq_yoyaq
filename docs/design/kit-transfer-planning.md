# キット移動計画 再設計 仕様（ドラフト）

作成: 2026-06-23 / 対象: `src/utils/kitOptimizer.ts`（純関数）＋ 移動計画タブ
状態: **要件すり合わせ中**（オーナー確認待ちの項目あり＝末尾「未決」）

## 目的

公演スケジュールから、各キットを「どの店舗に・いつ運ぶか」を計算する。
**最優先は計算の正しさ**（不足の見落とし／無駄な移動／早すぎる運び出しを出さない）。
移動日は固定設定ではなく **システムが最適提案** する。表示の形は別途決める。

## 合意済みの方針（2026-06-23）

| 項目 | 決定 |
|---|---|
| 最適化の目的 | **移動「回数」を最小**（＝動かすキット数を最小） |
| マルチホップ | **条件つきで複数回OK**（実行可能なら週内に何度でも A→B→A 等。回数は最小化）。2026-06-23 改定：当初「原則1週1回」としたが、ホーム店も普段使うため大量の偽不足が出たため見直し。条件＝下記「制約」を満たす移動のみ |
| 同日の扱い | 基本は 1キット=同日1拠点。**ただし同一拠点（同住所）なら時間が重ならなければ同日使い回しOK** |
| 移動日 | システムが提案 |
| 最優先 | 計算の正しさ |
| 表示 | 未定（後で一緒に決める） |

## データモデル

- **キット** = シナリオ(`scenario_master_id`) × キット番号(`kit_number`)。各キットは「その日どこか1つの拠点グループに居る」。
- **拠点グループ** = `kit_group_id` でまとめた店舗群（＝同住所・同一拠点とみなす。既存の仕組み）。同グループ内はキットを共有（移動不要）。
- **公演（需要の素）** = `{ date, store_id, scenario_master_id, start_time, end_time }`。
  - 時刻は `scheduleApi.getByDateRange` が返す（`start_time`/`end_time` 実装済み・確認済み）。
  - ※現行 `demands` は時刻を捨てているので、**時刻を載せるよう拡張する**。
- **必要キット数**（ある拠点グループ・ある日・あるシナリオ）= **時間帯が重なる公演の最大同時数**（区間の最大オーバーラップ）。
  - 例: 同住所で 10:00-13:00 と 14:00-17:00 → 重ならない → **1キットで両方カバー**。
  - 例: 同住所で 10:00-13:00 と 12:00-15:00 → 重なる → **2キット必要**。
  - 現行は「公演数ぶん必要」と数えており、ここが正しさのズレ。

## 固定（キット番号ごと・2026-06-23 決定）

- **固定は「キット番号ごと」**（`scenario_kit_locations.is_fixed`）。そのキットは移動計画で動かさない。
  同じシナリオでも #1 は固定・#2 は可、が可能。
- **店舗固定（`stores.kit_fixed`）は廃止**（planner では使わない）。
  ※既存の「店舗別在庫」タブの店舗ロックは撤去予定。
- planner は `fixedKitKeys: Set<\`${scenario_master_id}-${kit_number}\`>` を受け取り、該当キットを移動候補から除外。
  固定キットしか無い拠点に在庫がある場合、不足理由は `locked_fixed`（在る店を `lockedStoreIds` で示す）。
- 段取り: ① DB `scenario_kit_locations.is_fixed` 追加 → ② API（取得に is_fixed／トグル）→ ③ UI（現在の配置タブでキットごとにロック）。

## 制約（正しさの核）

1. キットは各日ちょうど1拠点グループに居る。
2. 移動は **実行可能なら複数回OK**（A→B→A 等の行き来可）。ただし回数は最小化（意味のない往復はしない）。
3. **公演で使う日は運び出さない**。移動（到着）日 T は「home での最終使用日より後」かつ「**dest 初回使用日の前日まで**」。
   - **前日必着（2026-06-23 決定）**：当日着は危険なので禁止。**T ≤ dest 初回公演日 − 1**。
   - **中日チェック**：T〜公演日の間に home でそのキットが必要なら、その移動は不可（現行は欠落）。
4. **固定店舗**（`kit_fixed`）は供給源にしない（出さない）。
5. **未配置キット**（`scenario_kit_locations` に無く現在地不明）は計画対象外（先に初期位置登録が必要）。

## 目的関数

1. **第一目標**：移動するキット数を最小（週1回前提なので「移動回数」と等価）。
2. **同点時の優先（tie-break）**：
   - (a) 近い拠点からの移動を優先（既存: region 一致 → 住所近接）
   - (b) 移動日はまとめやすい日／公演直前寄り　※どちらを上にするかは未決
3. 満たせない需要は**不足として明示**（後述）。

## 出力

- **移動リスト**：`(scenario, kit_number, from_store, to_store, transfer_date, 対象公演)`
- **解消できない不足**：拠点・日・シナリオ・不足数を明示（**現状はサイレント。必ず出す**）
- **移動日サマリ**：実際に移動が発生する日（例「この週は 6/27・6/29 に運搬」）

## アルゴリズム方針

シナリオごとに独立（別シナリオのキットは干渉しない）。各シナリオで:

1. 公演を (拠点グループ × 日) に集約し、各セルの **必要数＝最大同時数** を算出。
2. 在庫キット（home 位置）で「動かさず満たせる需要」を確定（移動0）。
3. 残る不足を、余剰キットの **1回移動** でカバー。移動可能性は制約2・3で判定。
4. **移動キット数が最小**になる割当を選ぶ（規模が小さいので厳密 or 高品質ヒューリスティック）。
5. 各移動の **T（移動日）** はシステムが決定：home 最終使用後 〜 **dest 初回公演日の前日まで**（T ≤ 初回公演−1・当日着禁止）の範囲で選ぶ。
6. 計算後に **`validateTransferPlan` を必ず実行**し、二重使用・残不足を検出して出力に反映（現状この検証関数は未使用）。

## テスト計画（`npm run test:unit`・kitOptimizer は純関数）

- 動かさず満たせる → 移動0
- 単純1移動でカバー
- 同住所・時間**非重複** → 1キット使い回し（移動0・不足なし）
- 同住所・時間**重複** → 2キット必要（在庫なければ移動 or 不足）
- home が中日に必要 → その移動は不可（別キット or 不足）
- **前日必着**：公演当日着になる移動は出さない（T ≤ 公演日−1）。前日着まではOK
- 固定店舗は供給しない
- 未配置キットは候補にしない
- 解消不能 → **不足として出る**（サイレントにしない）
- 同点で2案 → **移動回数が少ない方**を選ぶ
- 移動日提案が制約（home最終使用後〜dest初回使用前）を満たす

## 決定事項（2026-06-23・全確定）

1. ✅ **「同住所」の判定** = 既存 `kit_group_id`（手動グループ）を同一拠点とみなす。実住所の自動比較は足さない。
2. ✅ **tie-break の優先** = ①移動回数最小 → ②近い拠点（region→住所近接）→ ③公演前日寄りの日。
3. ✅ **「週1回」の例外** = 例外許可せず、週2回要るケースは**不足として明示**（正しさ優先）。必要なら後日 週2回解禁を検討。
4. ✅ **前日必着** = 当日着は禁止。**T ≤ 公演日 − 1**。

### 残課題（将来・今は入れない）
- 移動日に「運べない曜日（スタッフ都合等）」の制約があるか。今は home最終使用後〜公演前日の範囲で**任意日**に運べる前提。必要になれば「許可日フィルタ」を後付けする。

## API 契約（新モジュール `src/utils/kitTransferPlanner.ts`）

旧 `kitOptimizer.calculateKitTransfers` は当面据え置き、新ロジックは別モジュールで作りテストで固める。

※ `KitDemand` は `src/types/kit.ts` に別用途の同名型があるため、planner 入力は `PlannerDemand` とする。

```ts
export interface PlannerDemand {
  date: string              // 'YYYY-MM-DD'
  store_id: string
  scenario_master_id: string
  start_time: string        // 'HH:MM'（時間重複判定に使用）
  end_time: string          // 'HH:MM'
}

export interface KitShortageItem {
  date: string
  store_id: string
  scenario_master_id: string
  needed: number            // 必要数（最大同時数）
  available: number         // 確保できた数
}

export interface KitTransferPlan {
  transfers: KitTransferSuggestion[]   // transfer_date = 到着日（= 公演日-1 以前）
  shortages: KitShortageItem[]         // 解消できなかった不足（必ず返す）
}

// 移動日は引数で受け取らない（システムが最適日を決める）。
// today は「手遅れ判定」に必須：移動(到着)日 T は today <= T <= 公演日-1 の範囲のみ。
// この範囲が空（公演が今日以前 等）＝手遅れ → shortages に出す。
export function planKitTransfers(
  initialState: KitState,
  demands: PlannerDemand[],
  scenarios: Scenario[],
  stores: Store[],
  today: string,          // 'YYYY-MM-DD'
): KitTransferPlan
```

## 時間の基準・期間（2026-06-23 決定）

- **基準 = 今日**。**今日〜+14日**の公演を分析（過去日は出さない）。横軸の長さは将来変更可。
  - ※現行は「今週(月〜日)・過去日込み・週送り」で、“今ヤバい”が埋もれる原因だった。
- **移動可能日** = `today` 以降のみ（過去には運べない）。前日必着と合わせ `today <= T <= 公演日-1`。

## 状況把握（UI: 一覧ステータス ＋ 緊急ボード）

planKitTransfers の出力から導出する（別ロジック不要）：

- **一覧の行ステータス（シンプル2段階）**
  - ✅ **在庫あり** … その公演が transfers にも shortages にも無い（既にその拠点にキットがある）
  - ⚠️ **要対応** … その公演に対する移動が transfers にある（今は無いが手配すれば間に合う）
- **🔴 緊急ボード（最上部）＋該当行も赤**
  - (a) **手遅れ** … shortages に出た公演（今日以降に間に合わせる移動が無い／手段なし）
  - (b) **持ち越し** … 未実行の確定移動（`transferEvents` で到着予定日を過ぎても delivered になっていないもの）
  - 緊急ボードに出たものは一覧の該当行も赤で示す（ボード＋行色の両方）。

## 持ち越しの責任追及（未実行の確定移動・2026-06-23 追加）

過去に運べていなかった確定移動を、後から追跡できるように出す。

- **未実行の定義** = `kit_transfer_events` で `status='pending'` かつ `transfer_date < today` かつ **設置未完了**（対応する `kit_transfer_completions` に `delivered_at` が無い）。
- **出せる情報**（既存データで可能）:
  - 移動内容（シナリオ / キット番号 / from→to）
  - 予定日（`transfer_date`）と **超過日数**（today − transfer_date）
