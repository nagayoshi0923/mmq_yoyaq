# 新規訪問ユーザー視点 UXレビュー（MMQ予約サイト）

作成日: 2026-01-15  
前提: **既存のイシューリストは参照せず**、コードから読み取れる導線を「初見ユーザーが実際に体験する」前提で網羅的に洗い出した。

## 対象の代表導線（想定）

- **プラットフォームトップ**: `/`
- **シナリオ検索（プラットフォーム）**: `/scenario`
- **団体（組織）予約サイトトップ**: `/{organizationSlug}`（例: `/queens-waltz`）
- **団体（組織）シナリオ詳細**: `/{organizationSlug}/scenario/{scenarioId}`
- **予約確認（フォーム）**: シナリオ詳細内の「予約確認へ進む」→ `BookingConfirmation` 表示
- **静的ページ**: `/faq`, `/guide`, `/terms`, `/privacy`, `/contact`, `/cancel-policy`, `/stores`
- **組織別お問い合わせ**: `/org/{organizationSlug}/contact`

## イシューリスト（新規作成）

表記:
- **優先度**: P0=致命（予約できない/誤予約/情報漏洩/運用破綻）, P1=大（CV低下/問い合わせ増）, P2=中（迷う/不安）, P3=小（改善余地）
- **観点**: 初見ユーザーの迷い・不安・失敗・再訪時の混乱を重視

---

## 1) 情報設計・ナビゲーション

### UX-001: 「MMQ（プラットフォーム）」と「各団体（組織）」の切り替えが分かりにくい
- **優先度**: P1
- **再現**: `/`→ 店舗カードやシナリオから `/{slug}` に遷移 → ヘッダー左の `MMQ` を押すと `/` に戻る
- **困りごと**: 「今どこのサイトを見ているのか」（MMQ全体 vs 団体ページ）が初見だと判別しづらい。戻る操作が「前の画面」ではなく「別サイト（プラットフォームトップ）」になり、探索が途切れる。
- **期待**: パンくず/ラベル/説明でコンテキストが明確、戻るが期待通りに動く
- **提案**:
  - ヘッダーに「現在の団体名 + 予約サイト」等の明示
  - 「前に見ていた団体へ戻る」導線（履歴/直近訪問団体）をUIで提示

### UX-002: 組織スラッグが存在しない場合のリカバリ導線が不自然
- **優先度**: P1
- **再現**: 存在しない `/{slug}` にアクセス → 「組織が見つかりません」→ ボタンが `/queens-waltz` 固定
- **困りごと**: 初見ユーザーにとって `queens-waltz` は無関係な可能性が高く、「なぜ別団体に飛ばされるのか」不信感になる。
- **期待**: **プラットフォームトップ（`/`）** or **団体一覧（`/stores`）**へ案内
- **提案**: 「団体を探す」ボタンを `/stores` へ、もしくは `/` へ戻す

### UX-003: URLがハッシュ方式とパス方式の混在に見える（初見には不可解）
- **優先度**: P2
- **再現**: 旧URL（`/#booking/...` 等）を踏んだ場合にパスへ変換される実装がある
- **困りごと**: ブックマーク/共有URL/検索結果で複数形式が混在すると、ユーザーが「正しいURL」を判断できない。SNSで共有時の見た目も悪い。
- **期待**: パス方式に統一、旧URLはサーバ/クライアントで透過的に301/replace
- **提案**: 旧形式の露出を減らす（テスト/ドキュメント含む）

### UX-004: ヘッダーの遷移が `window.location.href` 依存で、体感が「別ページに飛ばされた」ように感じる
- **優先度**: P2
- **再現**: ヘッダーの `MMQ` / 団体名 / マイページ / ログイン等をクリック
- **困りごと**: SPAとしての遷移（即時性・状態保持）を損ね、戻る・スクロール位置・表示のチラつきが起こりやすい。
- **期待**: ルーター遷移（`navigate`/`Link`）でスムーズに移動

## 2) シナリオ閲覧・検索（予約前の情報収集）

### UX-005: 団体トップの「店舗フィルタ初期値=全店舗（臨時会場除外）」が気づきにくい
- **優先度**: P2
- **再現**: `/{slug}` 初回アクセス → 店舗フィルタが暗黙に全選択（臨時会場は除外）
- **困りごと**: 「表示されない店舗/公演がある」理由が分かりにくく、欠落バグに見える可能性がある。
- **期待**: 初期選択の意図がUIから分かる（例: フィルタに「◯店舗選択中」ラベル）
- **提案**: 初回は「全店舗」表示のトグル/説明、または臨時会場も含める/明示的に除外表示

### UX-006: ローディング中のログ計測が多く、体感/信頼に悪影響の可能性
- **優先度**: P3
- **再現**: 団体トップでデータロード時に複数回のレンダリング監視ログを出力
- **困りごと**: 本番でコンソールが騒がしいと不安（技術者ユーザー）・デバッグ情報漏洩・性能悪化の懸念。
- **期待**: 本番では最小限のログ

### UX-007: シナリオ詳細の「戻る」が履歴依存で、意図した場所に戻らない場合がある
- **優先度**: P2
- **再現**: `/{slug}/scenario/{id}` を外部から直リンク → 戻る（`navigate(-1)` 相当）
- **困りごと**: 外部流入だと「前のサイト」へ戻るだけになり、団体トップへ戻れない。迷子になりやすい。
- **期待**: 直リンク時は `/{slug}` へ戻る（フォールバック）

## 3) 予約フロー（ログイン/入力/確定）

### UX-008: 予約にログイン必須だが、理由・メリットが弱く初見にはハードル
- **優先度**: P1
- **再現**: シナリオ詳細で日程を選ぶ → 「ログインして予約する」
- **困りごと**: 「なぜログインが必要か」「ログインすると何ができるか」が伝わらず離脱しやすい。
- **期待**: ログイン必須の理由（予約確認・変更・キャンセル・重複防止等）を明示
- **提案**: ログイン誘導に説明文/所要時間/ゲスト予約可否の方針を追加

### UX-009: 予約人数のUIが共通UI部品ではなく生`<select>`で、見た目・操作感が不揃い
- **優先度**: P3
- **再現**: シナリオ詳細の予約パネルで人数選択
- **困りごと**: デザイン統一感が崩れる。モバイルでの操作性/アクセシビリティも不安。
- **期待**: プロジェクト標準の Select コンポーネントに統一

### UX-010: 空席より多い人数が選べてしまい、後段で弾かれる（体感が悪い）
- **優先度**: P2
- **再現**: 残席が少ない公演でも「最大人数」まで人数選択が可能 → 次で警告
- **困りごと**: 「選べたのに次で怒られる」体験は不満につながる。
- **期待**: 人数選択肢を「残席」上限に合わせる/選択後すぐに説明

### UX-011: 予約確認画面で「ログインが必要です」と出る可能性があり、導線が二重
- **優先度**: P2
- **再現**: 予約確認内でも未ログインを弾く実装がある
- **困りごと**: 予約確認へ進めたのに「ログイン必要」と言われると混乱（本来は前段で必ずログインさせるべき）
- **期待**: 予約確認表示前に必ずログイン状態が保証される

### UX-012: 予約完了後、次の行動（マイページで確認/キャンセル方法/当日の案内）が弱い
- **優先度**: P1
- **再現**: 予約完了表示 → 3秒後に自動で戻る、または「トップへ戻る」
- **困りごと**: 初見は「予約できたか不安」「予約内容を確認したい」「キャンセル方法は？」が自然に起きる。現状は次の導線が薄い。
- **期待**: 「予約内容を見る（マイページ）」「キャンセルポリシー」「当日の持ち物/集合」などの次アクション提示
- **提案**: 完了画面に CTA を追加、自動遷移は無効化/ユーザー選択に変更

### UX-013: 重複予約の警告文言がやや矛盾して見える
- **優先度**: P2
- **再現**: 重複予約警告で「人数を増やす場合はマイページから変更できます」/ 別箇所で「キャンセルしてマイページから変更」
- **困りごと**: 初見ユーザーは「変更できるの？できないの？」が分からず不安。
- **期待**: 1つの正しい手順に統一（変更可なら「変更」、不可なら「キャンセル→再予約」）

## 4) 満席時のキャンセル待ち

### UX-014: 満席時のキャンセル待ちもログイン必須で、初見には厳しい
- **優先度**: P2
- **再現**: 満席の公演 → キャンセル待ち登録 → ログインが必要
- **困りごと**: 「席がないのに登録のためにアカウント作成」は離脱理由になりやすい。
- **期待**: ゲスト登録の可否検討 or ログイン必須の理由明示（悪用対策等）

### UX-015: キャンセル待ちの「希望人数」の上限/妥当性が分かりにくい
- **優先度**: P3
- **再現**: 希望人数が 1〜10 で固定
- **困りごと**: 公演の最大人数と関係が見えない（最大が8なのに10まで選べる等の可能性）
- **期待**: 公演の最大人数・現実的範囲に連動

---

## 5) お問い合わせ・サポート

### UX-016: お問い合わせ送信完了後の「戻り先」が `/` 固定で、文脈が切れる
- **優先度**: P2
- **再現**: `/contact` から送信完了 → 「トップページへ戻る」→ `/`
- **困りごと**: 団体ページから来たユーザーは「元の団体予約サイト」に戻りたいのに、プラットフォームに戻される。
- **期待**: 直前のページ/団体トップへ戻れる（もしくは選択肢）
- **提案**: `organizationSlug` を持つ場合は `/{slug}` へ、なければ `/` へ

### UX-017: 団体別問い合わせ（`/org/{slug}/contact`）とMMQ問い合わせ（`/contact`）の違いが初見には不明
- **優先度**: P2
- **再現**: フッターには両方が存在（組織ページの法的リンクにMMQ問い合わせもある）
- **困りごと**: 「どこに問い合わせが届くのか」不安。誤送信/たらい回しの原因。
- **期待**: ラベルを明確化（例:「店舗へ問い合わせ」「MMQ運営へ問い合わせ」）

---

## 6) アクセシビリティ/モバイル

### UX-018: アイコンのみ（×）ボタンに aria-label がなく、支援技術で不親切
- **優先度**: P3
- **再現**: 貸切候補の削除ボタン等、視覚的に「×」のみ
- **困りごと**: スクリーンリーダーで意味が伝わらない
- **期待**: `aria-label="候補を削除"` 等を付与

### UX-019: `position: sticky` の検索バー/ヘッダーが重なりやすく、画面占有が大きい
- **優先度**: P3
- **再現**: 団体トップでヘッダー + 固定検索バー
- **困りごと**: モバイルでコンテンツ領域が狭くなる。スクロール時の体験が窮屈。
- **期待**: スクロール時に圧縮/折りたたみ等

---

## 7) QA・品質（初見体験に直結するもの）

### UX-020: E2Eが旧URL（ハッシュ形式）を前提としていて、実導線の退行を検知しにくい
- **優先度**: P2
- **再現**: Playwright が `/#booking/...` でアクセスする
- **困りごと**: 本来のURL（`/{slug}`）の壊れがテストで検知されない可能性。
- **期待**: パス形式のURLでテストする

---

## 付記（観察メモ）

- 予約確認・キャンセル待ちなどは「ログイン必須」方針が一貫している一方、初見ユーザーに対しては説明が薄いので、**コンバージョン最大のボトルネック**になりやすい。
- 組織スラッグが絡むページでは「戻る」「問い合わせ」の戻り先が文脈依存になるため、**フォールバック設計**が重要。
