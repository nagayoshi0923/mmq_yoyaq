## 第1弾: 削除操作の分割（コミット 8ef79605）

分割したコード: 削除確認ダイアログ／削除実行／直接削除。
分岐は5つ。**D-1 〜 D-5 を順に**実施してください（所要 約10分）。

### 準備
1. スケジュール管理ページを開く（管理画面 → スケジュール）
2. テスト用の通常公演を1つ作る:
   空きマスをクリック → 公演追加モーダル → シナリオ・GM・時間を適当に設定 → 保存
   （名前が分かりやすいシナリオ推奨。以後「テスト公演A」と呼ぶ）

### D-1: 予約ゼロの通常公演を削除（基本の削除）
| 手順 | 期待される結果 |
|---|---|
| ① テスト公演Aのマスを右クリック → 「公演を削除」 | 削除確認ダイアログが開く |
| ② ダイアログで削除を確定 | ダイアログが閉じ、**マスから公演が即座に消える** |
| ③ ページをリロード | **リロード後も消えたまま**（DBから本当に消えている） |
| ④ 該当マスをクリック → 履歴タブ（あれば） | 削除の履歴が記録されている |

### D-2: 有効な予約がある公演の削除は「拒否」される（最重要分岐）
| 手順 | 期待される結果 |
|---|---|
| ① テスト公演Bを新規作成（予約受付ON） | — |
| ② 予約サイト側からその公演に1件予約を入れる（自分のテスト顧客でOK） | 予約完了 |
| ③ スケジュール表に戻り、公演Bを右クリック → 「公演を削除」 → 確定 | **削除されない**。オレンジの警告トースト:「この公演には1件の有効な予約が紐付いているため削除できません」＋「代わりに『中止』機能を…」の案内 |
| ④ 画面と、リロード後の画面 | **公演Bは残ったまま**（予約も無事） |

### D-3: キャンセル済み予約「だけ」なら削除できる
| 手順 | 期待される結果 |
|---|---|
| ① D-2 の予約を予約一覧モーダルからキャンセルする | 予約ステータスがキャンセルに |
| ② 公演Bを右クリック → 「公演を削除」 → 確定 | **今度は削除に成功**（キャンセル済み予約は削除を妨げない） |
| ③ リロード | 消えたまま |

### D-4: 貸切公演の削除（公演は消え、申込は「キャンセル」で台帳に残る）
※ 2026-06-12 仕様変更: 申込の物理削除をやめ、キャンセル状態での保持に変更（顧客記録の保全）
| 手順 | 期待される結果 |
|---|---|
| ① 貸切申込→承認で貸切公演を1つ作る（消してよいもの！） | スケジュール表に貸切公演が表示される |
| ② その貸切公演を右クリック → 「公演を削除」 → 確定 | 公演がマスから消える（警告トーストは出ない） |
| ③ 貸切管理ページを開く | **申込は残っていて、ステータスが「キャンセル」になっている** |
| ④ 該当マスの更新履歴（公演追加→更新履歴タブ） | **「削除」の履歴が記録されている**（作成履歴だけでなく） |
| ⑤ リロード | 公演は消えたまま・申込はキャンセルのまま |

### D-5: 予約一覧からの「直接削除」（確認ダイアログを経ない別ルート）
| 手順 | 期待される結果 |
|---|---|
| ① 貸切公演をもう1つ用意し、参加者（予約）を表示できる状態にする | — |
| ② 公演の予約者バッジ等から**予約一覧モーダル**を開く | 予約一覧が表示される |
| ③ 貸切の参加者を**全員キャンセル**する | 「全員キャンセルしたのでイベントを削除しますか」相当の確認が出る |
| ④ 削除を選ぶ | モーダルが閉じ、公演がスケジュール表から消える |
| ⑤ リロード | 消えたまま |

### 異常系（任意・余裕があれば）

  赤いエラートースト「公演の削除に失敗しました」が出て、ダイアログが閉じ、公演は残る

## 第2弾: 中止・復活操作の分割＋削除/中止フロー統一（2026-06-13 改訂）

UI が2026-06-13に大きく変わったため改訂。中止・削除とも、有効予約があると
**2ステップ確認**（(1/2)キャンセル確認＝件数＋予約者一覧 → (2/2)メール送信確認＝
理由編集＋送信チェックボックス＋実行まとめ）を通る。予約ゼロの中止は確認なしで即実行。
**C-0 〜 C-4 を順に**実施（所要 約10分）。メール確認があるので自分のメールで予約しておくこと。
※ 公演作成直後の数秒は保存中（仮ID）のため、操作すると「保存処理がまだ完了していません」
トーストが出る。数秒待って再試行すればよい（エラーではなく正常なガード）。

### C-0: 予約ゼロの公演の中止 → 復活（確認なし即実行の分岐）
| 手順 | 期待される結果 |
|---|---|
| ① 予約のないテスト公演を右クリック → 「公演を中止」 | **確認ダイアログなし**で即中止表示＋「公演を中止しました」トースト |
| ② 同じ公演を右クリック → 「公演を復活」 | 通常表示に戻る＋復活トースト。履歴に中止と復活が記録されている |

### C-1: 通常公演の中止（予約あり・キャンセルメールが飛ぶ最重要分岐）
| 手順 | 期待される結果 |
|---|---|
| ① テスト公演を作り、予約サイトから**自分のメールアドレスで**1件予約 | 予約完了メールが届く |
| ② スケジュール表で公演を右クリック → 「公演を中止」 | **(1/2) 予約のキャンセル確認**が開き、件数と予約者（名前＋メール）が見える |
| ③ 「次へ：メール送信の選択」→ (2/2) で理由を「テスト中止です」に書き換え → ☑送信のまま「メールを送信して中止」 | 処理中→成功トースト。公演が**中止表示**になる。削除はされない |
| ④ 自分のメールを確認 | **キャンセルメールが届き、【キャンセル理由】に「テスト中止です」が入っている** |
| ⑤ リロード | 中止表示のまま。予約管理では予約がキャンセル済みで残っている |

### C-2: 理由をデフォルトのまま中止（定型文の確認）
| 手順 | 期待される結果 |
|---|---|
| ① 別のテスト公演＋自分のメールで予約 → 右クリック → 中止 → (2/2) で**理由を編集せず**「メールを送信して中止」 | 中止される |
| ② メールを確認 | 「誠に申し訳ございませんが、やむを得ない事情により公演を中止させていただく…」の**定型文**が入っている |

### C-3: 中止の取り消し（復活）
| 手順 | 期待される結果 |
|---|---|
| ① C-1 で中止した公演を右クリック → 「公演を復活」 | 公演が通常表示に戻る |
| ② リロード | 復活したまま（※取り消しメールは送られない仕様。予約はキャンセルされたままなので注意） |

### C-4: 貸切公演の中止
| 手順 | 期待される結果 |
|---|---|
| ① 消してよい貸切公演を1つ用意（申込→承認） | — |
| ② 右クリック → 中止 → 2ステップ確認（申込者が一覧に出る）→ メールを送信して中止 | 貸切公演が中止表示になり、**申込者にキャンセルメール**が飛ぶ |
| ③ 貸切管理ページ | 該当申込がキャンセル扱いになっている |

### F-1: 有効予約のある公演の削除（中止＋セル削除の統合フロー）
| 手順 | 期待される結果 |
|---|---|
| ① テスト公演＋自分のメールで1件予約 → 右クリック → 「公演を削除」 | 削除確定モーダルは出ず、**(1/2) 予約のキャンセル確認**が開く |
| ② (2/2) → 「メールを送信して削除」 | キャンセルトースト→セル消滅。メール着信。予約はキャンセル済みで台帳残存。履歴に削除記録 |
| ③ （任意）別の公演で「メールを**送らずに**削除」 | メールが来ないこと以外は同じ |

### D-5 再確認: 予約一覧から全員キャンセル→削除（コード書き換え後の回帰確認）
| 手順 | 期待される結果 |
|---|---|
| ① 予約入りの公演 → 公演モーダルの予約者タブ → 全員を個別キャンセル | 全員キャンセル後「参加者がいなくなったため削除しますか」相当の確認 |
| ② OK | セル消滅・履歴に削除。追加の2ステップ確認は出ない（有効予約ゼロのため） |
